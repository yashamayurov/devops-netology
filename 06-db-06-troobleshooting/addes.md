# Ответы на замечания.
## Задача 2

Перед выполнением задания познакомьтесь с документацией по [Redis latency troobleshooting](https://redis.io/topics/latency).

Вы запустили инстанс Redis для использования совместно с сервисом, который использует механизм TTL. 
Причем отношение количества записанных key-value значений к количеству истёкших значений есть величина постоянная и
увеличивается пропорционально количеству реплик сервиса. 

При масштабировании сервиса до N реплик вы увидели, что:
- сначала рост отношения записанных значений к истекшим
- Redis блокирует операции записи

Как вы думаете, в чем может быть проблема?
```
Если правильно понял документацию, то проблема в том, что Redis "не успевает" удалять истекшие записи. И для решения этой проблемы возможно использовать механизм Redis software watchdog, который постоянно отслеживает состояние записей с заданным периодом.
Включить данный можно командой:
CONFIG SET watchdog-period 500
```
## Задача 4

Перед выполнением задания ознакомтесь со статьей [Common PostgreSQL errors](https://www.percona.com/blog/2020/06/05/10-common-postgresql-errors/) из блога Percona.

Вы решили перевести гис-систему из задачи 3 на PostgreSQL, так как прочитали в документации, что эта СУБД работает с 
большим объемом данных лучше, чем MySQL.

После запуска пользователи начали жаловаться, что СУБД время от времени становится недоступной. В dmesg вы видите, что:

`postmaster invoked oom-killer`

Как вы думаете, что происходит?

Как бы вы решили данную проблему?

```
Насколько смог понять:
Ошибка возникает по причине нехватке оперативной памяти. 
Необходимо проверить настройку PostreSQL, чтобы они соответствавли имеющимуся оборудованию (ресурсам).
При дальнейшем добавлении ресурсов, настройки нужно будет снова изменить.

Также к этому возможно произвести настройку ядра, чтобы OOM Killer не закрывал PostgreSQL: указать значение 2 для параметра vm.overcommit_memory и ядро не станет резервировать больше памяти, чем указано в параметре overcommit_ratio. Но на мой взгляд, такой споссоб нежелателен, поскольку не гарантирует исключения завершения PostgreSQL и не решит проблем с нехваткой памяти.
```
